# Rootless Podman Web App with Nginx and Ansible

## Goal
Run the Flask image with rootless Podman and manage it as a systemd user service through Quadlet. The setup provides:

- automatic restarts  
- dependency management  
- clean logs  
- rootless operation  

User lingering keeps the service running after logout and starts the user manager during boot.

## Creating a non‑privileged user

On Debian and Ubuntu, `useradd` provides the low-level account-management command while `adduser` wraps it with prompts and sensible defaults. I used `adduser` for `appuser` because the account needed a home directory and password ([comparison](https://phoenixnap.com/kb/useradd-vs-adduser)).

Run the account and package setup on the target host from an existing administrator account. `adduser` prompts for the new account details. Then open a fresh SSH or console login as `appuser`; a plain `sudo -iu` shell might not have the user systemd environment.

```bash
sudo apt update
sudo apt install -y podman uidmap slirp4netns passt
sudo adduser appuser
sudo loginctl enable-linger appuser

grep '^appuser:' /etc/subuid
grep '^appuser:' /etc/subgid
```

Both `grep` commands must return an assigned range. If either is empty, allocate an unused host-specific range as described in Target 2 before starting a rootless container.

## Containerising the web server

The application is a small Python script that returns “Hello, World!” over HTTP. I built it as `quay.io/YOUR_QUAY_USER/hello-world-webserver:1.0` and first ran it manually as `appuser`, mapping the target host's loopback port 8080 to the same port in the container. Replace `YOUR_QUAY_USER` before running these commands in a normal login shell as `appuser`:

```bash
podman pull quay.io/YOUR_QUAY_USER/hello-world-webserver:1.0
podman run -d --replace \
  --name web \
  -p 127.0.0.1:8080:8080 \
  quay.io/YOUR_QUAY_USER/hello-world-webserver:1.0

```

The `curl` command returned **Hello, World!**, verifying that the container was running.

Remove the manual test container before starting the Quadlet, because both publish host port 8080:

```bash
podman rm -f web
```

## Trying the Quadlet approach

### Understanding Quadlet

Quadlet reads `.container` files from `$XDG_CONFIG_HOME/containers/systemd` or `~/.config/containers/systemd` and generates systemd services under `/run/user/<UID>/systemd/generator`. The generated units are transient, so the `.container` source file is what should be maintained rather than enabling the generated service directly ([Podman systemd unit documentation](https://docs.podman.io/en/latest/markdown/podman-systemd.unit.5.html)).

### Creating `.container` files

Testing Quadlet:

1. `web.container` (running the existing image on port 8080):
    
```ini
[Unit]
Description=Hello World Web Server (Quadlet)

[Container]
Image=quay.io/YOUR_QUAY_USER/hello-world-webserver:1.0
ContainerName=web-quadlet
PublishPort=127.0.0.1:8080:8080

[Service]
Restart=always
RestartSec=2

[Install]
WantedBy=default.target
```

  

The file went into `~/.config/containers/systemd/web.container`. After `systemctl --user daemon-reload`, the generator created `web.service` under `/run/user/<UID>/systemd/generator/`. The `[Install]` section lets the generator arrange startup under `default.target`; generated services themselves are not enabled with `systemctl enable`.

```bash
systemctl --user daemon-reload
systemctl --user start web.service
systemctl --user status web.service
```



### Dry‑run and manual promotion

The generator's `--dryrun` option prints the service file without installing it:

```bash
/usr/lib/systemd/system-generators/podman-system-generator --user --dryrun
```

The output includes the generated `[Unit]`, `[X-Container]`, and `[Service]` sections along with the final `podman run` command.

## Setting up an Nginx reverse proxy

The container listens on port 8080, but the lab required normal HTTP and HTTPS URLs. Nginx therefore listens on ports 80 and 443, redirects HTTP to HTTPS, terminates TLS, and proxies requests to `127.0.0.1:8080`. The `proxy_set_header` directives preserve the original host and client details ([Nginx reverse proxy guide](https://docs.nginx.com/nginx/admin-guide/web-server/reverse-proxy/)).

### Options considered

Two approaches were evaluated for the reverse proxy:

1. **System-wide Nginx (Option A)** – Install Nginx with `apt` and configure it to listen on ports 80 and 443. Proxy all requests to `http://127.0.0.1:8080`, add TLS by generating a self-signed certificate, and redirect HTTP to HTTPS. This method requires root privileges for installation and binding to privileged ports but integrates with system package management and security practices. Nginx’s documentation shows that an HTTPS server must specify `ssl_certificate` and `ssl_certificate_key` files in the server block ([nginx.org](https://nginx.org/en/docs/http/configuring_https_servers.html)).
    
2. **Rootless proxy in a container (Option B)** – Run Nginx (or another HTTP server) as an unprivileged user in a Podman container. With the host's default privileged-port setting, the proxy would listen on high ports such as 8081/8443 and forward traffic to the backend on port 8080. A systemd user unit or Quadlet could manage this container. Lower ports are possible only after a deliberate host capability, sysctl, or firewall change, which this lab did not make.
    

### Chosen solution

Because the application had to use ports 80 and 443, I chose system-wide Nginx. Its `/etc/nginx/sites‑available/hello.local` configuration contains two server blocks:

- A **HTTP block** listening on port 80 that returns a `301` redirect to `https://$host$request_uri;` ensuring all traffic is encrypted.
    
- An **HTTPS block** listening on port 443 with the `ssl` parameter enabled and pointing to the certificate and key files. Inside the `location /` block, `proxy_pass http://127.0.0.1:8080;` forwards requests to the Podman container, and `proxy_set_header Host $host;` and `proxy_set_header X-Real-IP $remote_addr;` ensure that the backend sees the original host and client IP ([docs.nginx.com](https://docs.nginx.com/nginx/admin-guide/web-server/reverse-proxy/)).
    

On Debian and Ubuntu, a symlink in `sites-enabled` activates the file from `sites-available`. Run this part on the target host from an account with `sudo` access:

```bash
sudo apt update
sudo apt install -y nginx openssl
sudo install -m 0755 -d /etc/nginx/tls

sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/nginx/tls/hello.local.key \
  -out /etc/nginx/tls/hello.local.crt \
  -subj "/CN=hello.local" \
  -addext "subjectAltName=DNS:hello.local"
```

Create the virtual host without allowing the shell to expand Nginx variables such as `$host`:

```bash
sudo tee /etc/nginx/sites-available/hello.local >/dev/null <<'NGINX'
server {
    listen 80;
    listen [::]:80;
    server_name hello.local;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    listen [::]:443 ssl;
    server_name hello.local;

    ssl_certificate /etc/nginx/tls/hello.local.crt;
    ssl_certificate_key /etc/nginx/tls/hello.local.key;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
NGINX

sudo ln -sf \
  /etc/nginx/sites-available/hello.local \
  /etc/nginx/sites-enabled/hello.local
sudo nginx -t
sudo systemctl reload nginx
```
The `openssl` command generates a self-signed certificate and private key, while `nginx -t` checks the configuration before the reload. A hosts file entry `127.0.0.1 hello.local` was added on the test machine so that `hello.local` resolved locally. The certificate is not publicly trusted; clients need the certificate added to their trust store or must use `curl -k` for this isolated test.

### Verification and testing

`curl http://hello.local` returned a 301 redirect to HTTPS. With `-k` to accept the self-signed certificate, `curl https://hello.local` returned `Hello, World!`. The backend remained a user service on port 8080. A 502 response usually meant that service was stopped or listening on the wrong interface, which could be confirmed in `/var/log/nginx/error.log`.

### Problems encountered and lessons learned

The `appuser` account could not install Nginx or bind ports 80 and 443, so those steps required administrator access. A rootless proxy on high ports would work, but it would not meet the port requirement without extra capabilities or firewall rules. Systemd service enablement and Nginx site enablement are also separate mechanisms: systemd uses `systemctl`, while Debian's Nginx layout uses symlinks under `sites-enabled`. The backend listens only on loopback, so `proxy_bind` was unnecessary.

## Full Ansible Runbook

The first half records the manual setup. This section develops the same lab into a reusable Ansible role and keeps the problems encountered during the work.

## Environment and Goals

The Flask app returns _Hello World_ from a rootless Podman container. Nginx publishes it on ports 80 and 443. The Ansible version had six requirements:

1. **Rootless Podman and networking:** run the app as an unprivileged user such as `appuser`. Rootless networking uses helpers such as **slirp4netns** or **pasta** ([Podman documentation](https://docs.podman.io/en/stable/markdown/podman.1.html)).
    
2. **Systemd integration via Quadlet:** manage the container as a systemd service without root. The Podman Quadlet generator reads `.container` files placed under `~/.config/containers/systemd` and produces transient user units. Rootless Quadlet files must not define `User`, `Group`, or `DynamicUser`; the service runs under that user's manager ([quadlet(5)](https://docs.podman.io/en/latest/markdown/podman-systemd.unit.5.html)).
    
3. **HTTP to HTTPS:** redirect port 80 to port 443, then proxy the request to the container on localhost. Nginx's `return 301` directive handles the redirect ([rewrite module](https://nginx.org/en/docs/http/ngx_http_rewrite_module.html)).
    
4. **Self-signed TLS certificates:** create one certificate per domain with Ansible's `community.crypto` modules. The role generates the key first, then a CSR, and finally the certificate.
    
5. **Private registry login:** authenticate to Quay with `podman_login`, using credentials stored in Ansible Vault.
    
6. **Reusable automation:** support multiple hosts and applications through role variables, keep secrets in Ansible Vault, and make check mode useful.

The controller needs the `containers.podman` and `community.crypto` collections. Install them on the Ansible controller and record the resolved versions before the run; the original notes did not include collection pins:

```bash
ansible-galaxy collection install containers.podman community.crypto
ansible-galaxy collection list containers.podman
ansible-galaxy collection list community.crypto
```

The managed hosts need SSH access for the inventory user and passwordless or interactive `sudo` as appropriate. `YOUR_QUAY_USER` remains a placeholder, and `quay_token` must come from a separately supplied Ansible Vault file; no registry credential is included here.

## Building the Container Image

A minimal Flask app listens on port 8080 and returns _Hello World_. This is the same image used in the preceding lab:

```dockerfile
FROM docker.io/library/python:3.12-slim-bookworm
WORKDIR /app
COPY app.py /app/app.py
RUN pip install --no-cache-dir Flask==3.1.2
EXPOSE 8080
CMD ["python", "app.py"]
```

After building and testing the image locally, it was pushed to **Quay.io** as `quay.io/YOUR_QUAY_USER/hello-world-webserver:1.0`. The repository used in this lab was private, so the Ansible role logs in with a robot-account token supplied through Ansible Vault.

The role writes that login to the app user's `$HOME/.docker/config.json`, a persistent fallback auth file that Podman reads. The default `${XDG_RUNTIME_DIR}/containers/auth.json` would disappear when `/run/user/<UID>` is recreated. Vault protects the token in the Ansible project, while the deployed auth file still contains usable credential material and is restricted to mode `0600`.

## Rootless Podman and Quadlet

Rootless Podman uses a helper such as `slirp4netns` or `pasta` for networking. The host installs `uidmap`, `slirp4netns`, and `passt`, which provides `pasta`. In this lab, `pasta` performed better but failed when port 8080 was already occupied. Changing the port or falling back to `slirp4netns` fixed the conflict.

### Quadlet Configuration

Rootless users place Quadlet files in `~/.config/containers/systemd/`. The generator writes the resulting units under `/run/user/$UID/systemd/generator/`. These files must not set `User`, `Group`, or `DynamicUser`; the service already runs as the file owner ([quadlet(5)](https://man.archlinux.org/man/quadlet.5.en)).

The template used for each application in the role is:

```ini
[Unit]
Description={{ container_name }} (rootless via Quadlet)

[Container]
Image={{ container_image }}
PublishPort={{ publish_port }}

[Service]
Restart=always
RestartSec=2

[Install]
WantedBy=default.target
```

The `publish_port` variable includes `127.0.0.1`, so only the local Nginx process can reach the published backend. Quadlet owns the generated `podman run` lifecycle, so the template does not need an extra `--replace` argument.

Per-container ID isolation with `UserNS=auto` is available as a hardening option, but it requires a dedicated `containers:` subordinate-ID range in `/etc/subuid` and `/etc/subgid`; see the [Podman systemd unit documentation](https://docs.podman.io/en/latest/markdown/podman-systemd.unit.5.html). This lab ran with Podman's default rootless mapping.

## TLS Certificate Generation

Ansible creates `/etc/ssl/{{ domain }}`, generates a private key, and builds a CSR containing the Common Name and SAN. The `x509_certificate` task signs the CSR with the `selfsigned` provider. `selfsigned_not_before` and `selfsigned_not_after` set the validity period. The crypto tasks are skipped in check mode because their output files do not yet exist.

## Nginx Reverse Proxy

Nginx terminates TLS and proxies HTTP requests to the rootless container. Each application gets two server blocks: one for HTTP-to-HTTPS redirection and one for TLS termination with proxying. The HTTP block issues a 301 redirect via the `return` directive ([nginx.org](https://nginx.org/en/docs/http/ngx_http_rewrite_module.html)), and the HTTPS block sets up TLS and proxies to the upstream port.

### Nginx Template

The Jinja2 template used by the role (`templates/nginx_server.j2`) is summarised below for `tls_mode: selfsigned`. The full template later in this document also handles `tls_mode: none` without redirecting clients to an HTTPS listener that does not exist.

```nginx
# HTTP → HTTPS redirect (always on)
server {
    listen 80;
    listen [::]:80;
    server_name {{ domain }};

    # Permanent redirect to HTTPS
    return 301 https://$host$request_uri;
}

{% if tls_mode != 'none' %}
# HTTPS reverse proxy (only when TLS enabled)
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name {{ domain }};

    # --- TLS certs ---
    ssl_certificate     {{ tls_root }}/{{ domain }}/fullchain.pem;
    ssl_certificate_key {{ tls_root }}/{{ domain }}/privkey.pem;

    # --- TLS settings (sane defaults) ---
    ssl_session_timeout 1d;
    ssl_session_cache   shared:SSL:10m;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;


    location / {
        # Upstream to your app
        proxy_pass http://127.0.0.1:{{ upstream_port }};

        # HTTP/1.1 keep-alive for upstream
        proxy_http_version 1.1;
        proxy_set_header Connection "";

        # Preserve original host and client info
        proxy_set_header Host                $host;
        proxy_set_header X-Forwarded-Proto   $scheme;
        proxy_set_header X-Forwarded-For     $proxy_add_x_forwarded_for;

        # Optional timeouts (uncomment/tune as needed)
        # proxy_read_timeout  60s;
        # proxy_send_timeout  60s;
        # send_timeout        60s;
    }
}
{% endif %}

```

## Ansible Role Design and Playbook

The automation lives in a `rootless_webapp` role. A variable list describes each application's name, user, domain, image, ports, TLS mode, and optional hosts-file entry.

### Role Structure and Source Code

The role has this layout:




```text
roles/
└─ rootless_webapp/
   ├─ defaults/
   │  └─ main.yml  # Role variables (safe defaults: ports, domain, tls_mode, paths)
   ├─ handlers/
   │  └─ main.yml   # service reload/restart handlers (nginx, systemd --user)
   ├─ tasks/
   │  ├─ main.yml     # task entrypoint (includes: tls.yml, quadlet.yml, nginx.yml)
   │  ├─ tls.yml      # self-signed cert/key provisioning and prechecks
   │  ├─ quadlet.yml  # write *.container, daemon-reload, and start logic
   │  └─ nginx.yml    # install nginx, place vhost, test config, enable/reload
   └─ templates/
      ├─ nginx_server.j2 # Jinja vhost: HTTP→HTTPS + reverse proxy to upstream_port
      └─ quadlet_container.j2 # Jinja Quadlet .container
```
- `tasks/main.yml` is the **router**: it imports `tls.yml` → `quadlet.yml` → `nginx.yml`.
    
- `quadlet.yml` drops the **Quadlet** file into `~/.config/containers/systemd/`, triggers `systemctl --user daemon-reload`, and starts the unit.
    
- `tasks/main.yml` installs Nginx. `nginx.yml` renders and enables each site, and the handler validates with `nginx -t` before reloading.
    
- `handlers/main.yml` centralizes **reload/restart** so changes are batched and idempotent.
    
- `defaults/main.yml` holds **all variables** (e.g., `domain`, `upstream_port`, `tls_root`, `tls_mode`, `user`).
#### defaults/main.yml

The default variables define the TLS root directory and Nginx paths.
```yaml
---

# --- Web applications to deploy (each with image/domain/upstream info)
apps: []

# --- Base directory where TLS certs and keys reside
tls_root: "/etc/ssl"

# --- Nginx configuration directories (Debian/Ubuntu layout)
nginx_sites_available: "/etc/nginx/sites-available"
nginx_sites_enabled:   "/etc/nginx/sites-enabled"
```

#### handlers/main.yml

The validation handler runs before the reload because handlers execute in the order listed here.

```yaml
---

- name: Validate nginx configuration
  become: true
  ansible.builtin.command: nginx -t
  changed_when: false
  listen: Reload nginx

- name: Reload nginx
  become: true
  ansible.builtin.service:
    name: nginx
    state: reloaded
  listen: Reload nginx
```

#### tasks/main.yml

This task file orchestrates the deployment:

```yaml

      - apps | length > 0

  loop: "{{ apps }}"
  when: app.image is match('^quay\.io/')

  ansible.builtin.include_tasks: quadlet.yml
```

```yaml
---

# --- Define local TLS path facts ---------------------------------------------

- name: Set TLS paths for this app
  ansible.builtin.set_fact:
    _tls_dir: "{{ tls_root }}/{{ app.domain }}"
    _tls_key: "{{ tls_root }}/{{ app.domain }}/privkey.pem"
    _tls_crt: "{{ tls_root }}/{{ app.domain }}/fullchain.pem"

# --- Directory structure -----------------------------------------------------

- name: Create TLS directory
  become: true
  ansible.builtin.file:
    path: "{{ _tls_dir }}"
    state: directory
    owner: root
    group: root
    mode: "0755"

# --- Key and certificate generation (self-signed mode) -----------------------

- name: Generate private key (self-signed mode)
  when: app.tls_mode == 'selfsigned' and (not ansible_check_mode)
  become: true
  community.crypto.openssl_privatekey:
    path: "{{ _tls_key }}"
    size: 2048
    type: RSA
    mode: "0600"

- name: Generate CSR with SAN
  when: app.tls_mode == 'selfsigned' and (not ansible_check_mode)
  become: true
  community.crypto.openssl_csr:
    path: "{{ _tls_dir }}/{{ app.domain }}.csr"
    privatekey_path: "{{ _tls_key }}"
    common_name: "{{ app.domain }}"
    subject_alt_name:
      - "DNS:{{ app.domain }}"

- name: Generate self-signed certificate from CSR (valid 365 days)
  when: app.tls_mode == 'selfsigned' and (not ansible_check_mode)
  become: true
  community.crypto.x509_certificate:
    path: "{{ _tls_crt }}"
    csr_path: "{{ _tls_dir }}/{{ app.domain }}.csr"
    privatekey_path: "{{ _tls_key }}"
    provider: selfsigned
    selfsigned_not_before: "+0s"
    selfsigned_not_after: "+365d"

# --- Debug output when TLS disabled ------------------------------------------

- name: Note when TLS is disabled
  when: app.tls_mode == 'none'
  ansible.builtin.debug:
    msg: "TLS disabled for {{ app.domain }} (tls_mode=none)"
```

## Problems Encountered and Solutions

The lab exposed several problems worth recording:

1. **Deprecated parameters in crypto modules:** Early examples used parameters (`subject`, `dns_subject_alt_name`, `valid_days`) that were removed from the `x509_certificate` module. The role now generates a CSR with `openssl_csr` and uses `selfsigned_not_before`/`selfsigned_not_after` to set validity ([Ansible module documentation](https://docs.ansible.com/ansible/latest/collections/community/crypto/x509_certificate_module.html)).
    
2. **Check‑mode directory creation:** In check mode, key and certificate generation failed because the certificate directory did not exist. The role creates the directory and skips crypto tasks when `ansible_check_mode` is true.
    
3. **Missing `python3-apt`:** The `apt` module requires `python3-apt` to simulate changes in check mode. Installing this package via ad‑hoc Ansible resolved the issue.
    
4. **ACL errors:** When `podman_login` runs as an unprivileged user, Ansible sets ACLs on temporary files. Without the `acl` package the command failed with `chmod: invalid mode: 'A+user:appuser:rx:allow'`. Installing `acl` or setting `allow_world_readable_tmpfiles=True` in `ansible.cfg` fixed the problem.
    
5. **Non‑existent remote tmp directory:** Ansible uses `~/.ansible/tmp` to stage files. If the home directory is missing or unwritable, the playbook fails. Setting `ansible_remote_tmp=/tmp` in the inventory or `remote_tmp` in `ansible.cfg` ensured a writable location.
    
6. **Port conflicts:** Podman, especially with `pasta`, cannot bind a host port that is already in use. Freeing port 8080 or changing `upstream_port` to another value (e.g., `18080`) solved the issue.
    
7. **User login confusion:** Using `login` in scripts hung because it expects a tty. `sudo -iu appuser` is suitable for ordinary Podman commands, but it does not always create the runtime environment needed by `systemctl --user`. The role enables lingering and supplies `XDG_RUNTIME_DIR` and `DBUS_SESSION_BUS_ADDRESS` when managing the user service.
